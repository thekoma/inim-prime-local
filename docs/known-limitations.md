# Known limitations

Honest boundaries of this integration.

- **Output control needs the panel "Code".** Toggling panel outputs via the API requires the output-control Code to be configured on the panel; otherwise the panel rejects the command and Home Assistant raises a clear error. The output switches are therefore **hidden by default** — enable them only if your panel allows API output control.
- **Realtime push is a one-time manual setup.** Instant updates use the panel's event-action ("Invio pagine web") table, which you configure once in PrimeStudio (see [realtime.md](realtime.md)). There is no API to program those entries automatically, so without that setup the integration relies on its (already fast) adaptive polling.
- **Some commands depend on the panel/firmware.** On some panels `get_gsm_status` and `get_log_elements` return *not implemented*; the corresponding data (GSM diagnostics, event log) is then unavailable via the local API.
- **Scenario "active" state is only reliable for the *Total* macro.** The panel exposes a per-scenario `st` flag, but on a live PrimeX (fw 4.07) it only flips to "active" for the system-wide **Total** arm/disarm scenario. Single-area scenarios (e.g. *Ins.Box*, *Dis.Esterno*) never report themselves active even right after you run them — the panel keeps "Disarm Total" flagged until the whole system is armed. This was verified by sampling at ~10 reads/s across a full arm→disarm→arm cycle: the partition state changed correctly every time, the scenario flag never did. **Use the per-area `alarm_control_panel` entities as the source of truth for what is armed** — they are always correct. The `select` still *applies* scenarios reliably; only its displayed "current" value (and the per-scenario `binary_sensor`s, which are **disabled by default** for this reason) inherit the panel's limitation.
- **Sirens are not controllable.** The local API exposes panel **outputs** (`get_outputs_status`), but a siren is driven by the panel's own alarm logic and is not presented as a user-toggleable output — so there is no Home Assistant `siren` entity. What you *do* get is the siren's **anti-sabotage / supervision line** as a `tamper` `binary_sensor` (e.g. *AS Sirena Ext.*), and the *"alarm is sounding"* condition via the area `alarm_control_panel` going **triggered**. This is by design: a security alarm's siren should not be silenceable from an external integration.
- **Zone device class is a best-effort guess from the label.** Icons are derived from the zone name (window, door, garage, motion, smoke, gas, CO, water, vibration, tamper). Itʼs a heuristic — if a zone gets the wrong icon, override the **device class** on the entity in Home Assistant.
- **Renaming is forward-looking for `entity_id`s.** The panel **Name** (set at setup, editable via *Reconfigure*) and the per-room device grouping drive the **device and friendly names**. Home Assistant never auto-renames an existing `entity_id`, so a panel first added under an older version (when the entity_id was derived from the IP) keeps those `entity_id`s until you rename the entities — or remove and re-add the integration — even though the friendly names update immediately.
- **No automatic discovery.** INIM panels don't advertise themselves on the network, so you enter the host/port manually (no mDNS/SSDP/DHCP discovery).
- **Prime / PrimeX only.** This integration speaks the Prime on-board HTTP API. **SmartLiving / Nexus** panels use a different protocol — see the alternatives in the README.
- **Plaintext on the LAN.** The panel API is HTTP and the key is a bearer token; keep it on a trusted LAN segment with the panel's IP/MAC allow-list enabled, and never expose it to the internet (see the security notes in the README).
- **Fault decomposition is best-effort.** The per-fault binary sensors are derived from the panel's documented fault bitmap; the exact per-bit mapping is treated conservatively and may be refined as more real fault data is observed. The aggregate **System fault** sensor is the reliable headline.
- **In-Home-Assistant brand icon.** The custom INIM-style icon ships in this repo, but the icon shown *inside* Home Assistant comes from the official `home-assistant/brands` repository (a separate, core-only submission); until then HA shows a default icon.
